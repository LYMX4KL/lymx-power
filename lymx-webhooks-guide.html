Webhooks guide

1. Overview

LYMX webhooks deliver events to your server when things happen on your Business account. Events fire within 1-3 seconds of the underlying action (transaction, redemption, etc.).

Compared to Zapier: webhooks are direct, faster, free, but require code on your end. Most Businesses with internal engineers prefer webhooks; everyone else uses Zapier.

PrerequisitesYou need: (1) a public HTTPS endpoint on a server you control, (2) the ability to verify HMAC SHA-256 signatures, (3) basic familiarity with HTTP POST handling.

3. Configure a webhook

From your Business dashboard:

  1. Settings → API → Webhooks → "+ Add endpoint"
  2. Enter your HTTPS URL (e.g., https://api.yourbusiness.com/lymx/webhook)
  3. Select events to subscribe to (multi-select)
  4. Save → You'll receive a webhook secret — store it. Used for HMAC verification.
  5. Send test event → confirm your endpoint receives it (200 OK response expected)

4. Payload structure

All events POST a JSON body with this structure:

{
  "id": "evt_a1b2c3d4e5f6",           // unique event ID
  "type": "transaction.completed",    // event name
  "timestamp": 1714512345,                // unix epoch (UTC)
  "business_id": "biz_brewbean",         // your Business ID
  "data": {
    "transaction_id": "tx_xyz789",
    "customer_id": "cust_4827",         // anonymized wallet ID, no PII
    "amount": 2400,                       // USD cents · $24.00
    "lymx_issued": 120,                   // at 5/$1 rate
    "location_id": "loc_henderson",
    "items": [
      { "name": "latte", "price_cents": 600 },
      { "name": "breakfast burrito", "price_cents": 1800 }
    ]
  }
}

Headers on the POST request:

Content-Type: application/json
X-LYMX-Signature: sha256=a1b2c3d4...
X-LYMX-Event-Id: evt_a1b2c3d4e5f6
X-LYMX-Timestamp: 1714512345
User-Agent: LYMX-Webhooks/1.0

5. HMAC signature verification

Every webhook is signed with HMAC SHA-256 using your webhook secret. Always verify the signature — otherwise an attacker could forge fake events to your endpoint.

Compute expected signature: HMAC-SHA256(secret, raw_body_string). Compare with the X-LYMX-Signature header (strip the sha256= prefix). Use a constant-time string comparison.

Security warningDo NOT use == for signature comparison — it's vulnerable to timing attacks. Use crypto.timingSafeEqual in Node, hmac.compare_digest in Python, etc.

6. Retry policy

If your endpoint returns a non-2xx status (or times out after 10 seconds), LYMX retries with exponential backoff:

  • 1st retry: 30 seconds later
  • 2nd retry: 5 minutes later
  • 3rd retry: 30 minutes later
  • 4th retry: 4 hours later
  • 5th retry: 24 hours later
  • After 5 failures: webhook disabled, alert email sent to Owner

Each retry sends the SAME event (same X-LYMX-Event-Id), so your endpoint should be idempotent — handle duplicate event IDs gracefully.

7. Code examples

Node.js / Express

import express from 'express';
import crypto from 'crypto';

const app = express();
const WEBHOOK_SECRET = process.env.LYMX_WEBHOOK_SECRET;

// Use raw body for HMAC verification
app.post('/lymx/webhook', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-lymx-signature'].replace('sha256=', '');
  const expected = crypto
    .createHmac('sha256', WEBHOOK_SECRET)
    .update(req.body)
    .digest('hex');

  if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
    return res.status(401).send('Invalid signature');
  }

  const event = JSON.parse(req.body.toString());
  console.log('Received', event.type, event.id);

  // Handle the event (idempotent — check event.id against your DB)
  switch (event.type) {
    case 'transaction.completed':
      handleTransaction(event.data);
      break;
    case 'customer.first_visit':
      handleNewCustomer(event.data);
      break;
  }

  res.status(200).send('OK');
});

Python / Flask

from flask import Flask, request, abort
import hmac, hashlib, json, os

app = Flask(__name__)
WEBHOOK_SECRET = os.environ['LYMX_WEBHOOK_SECRET'].encode()

@app.route('/lymx/webhook', methods=['POST'])
def lymx_webhook():
    signature = request.headers.get('X-LYMX-Signature', '').replace('sha256=', '')
    expected = hmac.new(WEBHOOK_SECRET, request.data, hashlib.sha256).hexdigest()

    if not hmac.compare_digest(signature, expected):
        abort(401)

    event = json.loads(request.data)
    print(f"Received {event['type']} {event['id']}")

    # Handle the event
    if event['type'] == 'transaction.completed':
        handle_transaction(event['data'])

    return 'OK', 200

8. Testing

Two ways to test before going live:

Option A — Send test event from dashboard

Settings → API → Webhooks → tap your endpoint → "Send test event." Generates a synthetic transaction.completed with mock data. No real customer affected.

Option B — Use ngrok for local development

Tunnel localhost to public URL with ngrok http 3000. Use the ngrok URL as your webhook endpoint. Test against your local server. Disable when done.

$ ngrok http 3000
Forwarding  https://a1b2c3d4.ngrok.io -> http://localhost:3000

# Use https://a1b2c3d4.ngrok.io/lymx/webhook as your webhook URL

FAQ

What's the difference between webhook and Zapier?

Same events, different delivery. Webhooks: direct HTTP POST to your server, requires code. Zapier: visual no-code builder, slightly slower latency, costs $20+/mo. Use webhooks if you have a tech team, Zapier otherwise.

How many webhooks can I configure?

Up to 10 endpoints per Business. Each can subscribe to a different subset of events. Useful for routing different events to different services.

What if my server is down when an event fires?

LYMX retries 5 times over 24+ hours. After that, webhook is disabled + you get an email. You can re-enable from the dashboard once your server is back. Missed events during outage can be replayed from Settings → Webhooks → Replay events.

Are webhook payloads idempotent?

Each event has a unique X-LYMX-Event-Id. Retries reuse the same ID. Build idempotency by checking event ID against your DB before processing.

Can I subscribe to events for a specific location only?

For multi-location chains: yes, configure per-location filtering in webhook settings. Default: all locations on the Business account.

Are webhooks rate-limited?

1,000 events/day per Business with no extra charge. Heavy users (10k+/day) need an enterprise tier. We never throttle real-time delivery — we deliver immediately + log for billing.
