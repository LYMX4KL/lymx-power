December 18, 2025 · 19:42 UTC

Stripe webhook delays caused settlement statement generation to lag 14 minutes

⏱ 14 min lag 👥 4 Businesses saw delayed statement 📍 Settlement service
Minor

A burst of Stripe webhook events queued up faster than our consumer could process them, delaying the December 18 settlement statement generation. All amounts correct — just late by 14 minutes. No customer balance affected.

Show post-mortem

Timeline

19:30 UTC
Monthly settlement job scheduled to run. Queue size starts spiking.
19:34 UTC
Sentry alert: webhook consumer p99 latency over 60s.
19:38 UTC
Engineer paged. Identified Stripe API throttling from a non-rate-limited internal sweep.
19:42 UTC
Internal sweep paused. Webhook consumer drained the queue.
19:56 UTC
Settlement statements completed for all 4 affected Businesses. Email reminders sent.
Root cause

An internal data-cleanup script that calls Stripe was running on the same hour as monthly settlement. Stripe's per-account rate limit caused a thundering herd against our webhook consumer.

What's better now
  • All internal Stripe-touching scripts now use a separate API token with its own rate-limit budget.
  • The settlement job now runs at 03:00 UTC (off-peak) instead of 19:30 UTC.
  • Added a Grafana alert for "settlement job lag" that pages 5 minutes earlier than the prior alert.
November 4, 2025 · 14:17 UTC

Customer wallet QR codes failed to refresh for ~30 minutes

⏱ 31 min 👥 ~840 customers 📍 Wallet service
Major

A bad deploy to the wallet service caused QR code refresh requests to error out. Customers with already-loaded QR codes could still pay. New customers hitting the wallet got an error and had to refresh manually. No payments lost; some friction at the counter.

Show post-mortem

Timeline

14:10 UTC
Wallet service deploy completed. QR code refresh endpoint started returning 500s.
14:14 UTC
Sentry alert: 500 error rate spike on /wallet/qr/refresh.
14:17 UTC
Engineer paged. Confirmed correlation with the deploy.
14:22 UTC
Rolled back to the previous version. Error rate dropped to baseline within 60s.
14:30 UTC
Affected customers (estimate via failed-refresh count): ~840.
14:48 UTC
Service fully restored, status banner removed.
19:00 UTC
Notification email sent to all 840 affected customers + 50 LYMX inconvenience credit.
Root cause

The new deploy introduced a regression in how QR code expiry timestamps were serialized — the JSON serializer formatted the timestamp as ISO8601 instead of unix epoch, breaking the client-side expiry check. Caught in code review by one reviewer; a second reviewer didn't have the wallet client-side context to spot it.

What's better now
  • Added an end-to-end smoke test that hits /wallet/qr/refresh after every deploy, before traffic flips over.
  • Required code reviewers for wallet-service PRs now include at least one engineer with client-side context.
  • Deploys to the wallet service are now canary-rolled — 5% of traffic for 10 min before full rollout.
  • Auto-rollback triggers if 5xx rate spikes 10× baseline during canary.
